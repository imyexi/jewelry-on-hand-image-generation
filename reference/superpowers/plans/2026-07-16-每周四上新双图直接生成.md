# 每周四上新双图直接生成实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 为“每周四上新”视图中 13 个资料已录入商品自动选择浅色参考图，直接生成并质检上手图和生活场景图，将通过结果 URL 回填到原记录。

**Architecture:** 本批采用独立输出目录和批次清单，不修改项目技能的默认深色背景规则。飞书 Base 提供商品记录和附件，线上参考图库缓存提供角色明确的候选图，批次选择器仅在本次运行中忽略深色门禁并执行浅色门禁；AIReiter 负责两图输入生成，本地结构化 QC 决定是否回填。

**Tech Stack:** PowerShell、Python 3.11+、Pillow、`lark-cli base`、项目 `jewelry-on-hand-workflow` 契约、AIReiter `gpt_image_2`、JSON 审计文件。

## Global Constraints

- 所有测试、下载、候选、生成、QC 和回填证据只能写入 `output/2026-07-16-每周四上新双图/`。
- 所有长期说明和最终报告写入 `reference/`。
- 只处理 `SY1443`、`SY1444`、`SY1445`、`SY1446`、`SY1447`、`SY1448`、`SY1449`、`SY1450`、`SY1451`、`SY1452`、`SY1454`、`SY1456`、`SY1457`。
- 本批忽略现有技能的深色背景门禁，不修改技能默认行为；浅色透亮、白/绿色系、明亮高级、基础款/T 恤/素雅松弛穿搭是本批硬门禁。
- `图片类型` 仍是输出角色的唯一分类依据；`hand_worn` 不得使用生活场景图，`lifestyle` 不得使用手部佩戴图。
- 每个角色固定两张模型输入：参考图在前，产品上手/上身图在后；产品正面图只用于分析和 QC。
- 只有 `qc.status=pass` 的结果可以回填；目标字段非空、记录映射变化或回读不一致时停止对应记录。
- 不处理 56 条空白占位记录，不覆盖执行期间出现的已有值。

### Task 2: 自动选择浅色角色参考图

**Files:**
- Create: `output/2026-07-16-每周四上新双图/references/candidates.json`
- Create: `output/2026-07-16-每周四上新双图/references/decisions.json`
- Create: `output/2026-07-16-每周四上新双图/references/contact-sheets/<SKU>.jpg`
- Create: `output/2026-07-16-每周四上新双图/references/selected/<SKU>-<role>.<ext>`

**Interfaces:**
- Consumes: `batch.json`、`output/feishu_reference_cache/manifest.json` 和缓存图片。
- Produces: 每个未阻断 SKU 的 `hand_worn`、`lifestyle` 各一个已校验参考图及 Top 3 审计记录。

- [ ] **Step 1: 验证参考源完整性**

读取 `manifest.json` 和 `pending_enrichment.json`，要求 `pending_enrichment.records` 为空、所用记录 `usable=true`、图片路径存在且 SHA-256 可计算。保存 manifest SHA-256 到 `candidates.json`。

- [ ] **Step 2: 执行本批浅色门禁**

候选文本由 `关键词 + 风格分类 + 推荐使用方式 + 备注` 组成。必须命中至少一个浅色信号：`浅色`、`白色`、`绿色`、`自然光`、`清透`、`明亮`、`奶油`；必须排除：`暗调`、`深色背景`、`黑色背景`、`低照度`。服装优先 `白色上衣`、`白色衣袖`、`白色衬衫`、`T恤`、`针织`、`素雅`、`松弛`。

- [ ] **Step 3: 执行角色和品类门禁**

`hand_worn` 仅接受 `图片类型` 含“手部佩戴图”的记录；`lifestyle` 仅接受含“生活场景图”的记录。`适用品类` 必须包含当前产品形态对应品类或“通用”。戒指候选还必须有清楚手指和戒面空间，项链候选必须有颈胸或手持链条空间。

- [ ] **Step 4: 排序并保存 Top 3 与自动决策**

评分顺序为：角色匹配、品类匹配、浅色信号、白/绿色背景、素雅服装、主体展示面积、姿态自然、原参考首饰混入风险。保存 Top 3 后自动采用第一个通过视觉检查的 Rank；Rank 1 失败时依次检查 Rank 2、Rank 3。

- [ ] **Step 5: 视觉检查全部 26 个自动决策**

联系表每张显示商品上手图、Top 3 和自动采用标记。逐张确认背景明亮、服装符合要求、角色正确、佩戴空间足够；将结论、风险和选中参考图 SHA-256 写入 `decisions.json`。

### Task 3: 构建提示词并提交 26 个生成任务

**Files:**
- Create: `output/2026-07-16-每周四上新双图/runs/<SKU>/<role>/analysis.json`
- Create: `output/2026-07-16-每周四上新双图/runs/<SKU>/<role>/prompt.txt`
- Create: `output/2026-07-16-每周四上新双图/runs/<SKU>/<role>/submit.json`
- Create: `output/2026-07-16-每周四上新双图/runs/<SKU>/<role>/result.json`
- Create: `output/2026-07-16-每周四上新双图/runs/<SKU>/<role>/result.png`

**Interfaces:**
- Consumes: 商品证据图和 `decisions.json`。
- Produces: 最多 26 张待 QC 的本地结果和稳定 `out_task_id`。

- [ ] **Step 1: 为每个商品写产品保真分析**

`analysis.json` 必须记录产品形态、珠径或尺寸、圈数/层数/目标指位、颜色、透明度、材质纹理、主珠、隔珠、配件位置、`must_keep` 和 `must_not_change`。只记录图片可见事实，不依据材质名补造不可见结构。

- [ ] **Step 2: 构建角色专属 Prompt**

每份 Prompt 明确图 1 只提供人物、姿势、构图、浅色背景、素雅服装和光线；图 2 是产品身份唯一来源。共同要求为 3:4、2K、真实自然摄影、白/绿色系、明亮高级、无文字水印 logo、无额外首饰；`hand_worn` 强调真实佩戴，`lifestyle` 强调生活氛围且产品完整清晰。

- [ ] **Step 3: 校验两图输入和提示词编码**

逐任务验证参考图 SHA-256 与 `decisions.json` 一致、产品图 SHA-256 与 `batch.json` 一致、Prompt 为 UTF-8 且不含“深色背景”“暗调背景”等冲突词。

### Task 4: 严格 QC 与一次针对性重跑

**Files:**
- Create: `output/2026-07-16-每周四上新双图/runs/<SKU>/<role>/qc.json`
- Create: `output/2026-07-16-每周四上新双图/qc/contact-sheets/<SKU>.jpg`
- Create: `output/2026-07-16-每周四上新双图/qc/summary.json`

**Interfaces:**
- Consumes: Task 3 的输入证据、参考图、分析和生成结果。
- Produces: 每个角色唯一的 `pass`、`rerun` 或 `reject` 结论，以及可回填结果集合。

- [ ] **Step 1: 制作逐商品 QC 联系表**

每张联系表同时展示产品正面图、产品上手图、选中参考图、上手结果和生活场景结果，保留原始比例并标注 SKU、角色和任务 ID。

- [ ] **Step 2: 完整检查产品和人体**

逐项记录品类、数量、颜色、透明度、纹理、尺寸比例、圈数/层数/指位、主珠、隔珠、配件、接触阴影、重力、穿模、手指和肢体结构、浅色背景、素雅服装、额外首饰、文字、水印和 logo。任一关键结构错误不得 `pass`。

- [ ] **Step 3: 对首次可修复失败执行一次重跑**

从失败项生成针对性纠偏句，保持原产品图和选中参考图不变；保存新任务 ID 和结果，不覆盖第一次生成。严重改款、品类错误或第二次仍失败时标记 `reject`。

- [ ] **Step 4: 汇总可回填集合**

`summary.json` 按 SKU 和角色记录最终 generation、状态、任务 ID、本地文件、输出 URL 和失败原因。只有状态精确为 `pass` 且文件 SHA-256 匹配的条目进入 Task 5。
